# MiniLangVM: генерация кода и выполнение программы

## 1. Общая идея

В моём проекте основная часть выполнения программы сделана через собственный байткод и свою виртуальную машину.

То есть программа не выполняется сразу из AST и не переводится сразу полностью в машинный код. Сначала компилятор проходит несколько этапов:


исходный код
    -> лексер
    -> парсер
    -> абстрактное синтаксическое дерево
    -> семантический анализ
    -> генерация байткода
    -> выполнение на виртуальной машине


Главная часть проекта — это байткодная виртуальная машина MiniLangVM.

Ещё дополнительно у меня есть генерация x86-64 NASM ассемблера. Но это дополнительная возможность. Она покрывает не весь язык, а только часть. Полностью язык работает именно через VM.

## 2. Байткод

После семантического анализа программа уже проверена по типам. После этого она передаётся в генератор байткода.

Байткод в моём проекте состоит из:

* списка функций;
* списка инструкций;
* индекса функции main;
* информации о функциях: имя, параметры, тип возвращаемого значения, адрес начала функции.

Каждая инструкция содержит:

* код операции;
* аргументы инструкции;
* место в исходном коде.

Место в исходном коде нужно для ошибок времени выполнения. Например если деление на ноль произошло на строке 20, VM сможет вывести эту строку.

## 3. Виртуальная машина

Виртуальная машина у меня стековая.

Это значит, что большинство вычислений идут через стек.

Например выражение:

```
x + y
```

работает примерно так:

```
положить x на стек
положить y на стек
снять два значения
сложить
положить результат обратно на стек
```

Основные структуры VM:

* стек значений;
* стек вызовов функций;
* локальные переменные текущей функции;
* указатель на текущую инструкцию;
* таблица функций.

Когда вызывается функция, создаётся новый кадр вызова. В нём хранятся локальные переменные и параметры этой функции.

Аргументы передаются по значению. То есть функция получает свои значения, а не напрямую переменные вызывающего кода.

## 4. Значения во время выполнения

Во время выполнения все значения представлены специальной структурой Value.

Value хранит:

* тип значения;
* само значение;
* дополнительные данные для строк, массивов и структур.

Поддерживаемые значения:

* int;
* uint;
* float64;
* bool;
* string;
* array;
* struct;
* void.

Generic-типы не должны появляться как реальные значения во время выполнения. Они нужны на этапе семантического анализа. То есть T в generic-функции — это не отдельный runtime-тип, а placeholder, который проверяется компилятором.

## 5. main и код завершения

В программе обязательно должна быть функция:

```
int main()
```

Именно с неё начинается выполнение программы.

Значение, которое возвращает main, становится кодом завершения программы.

Например:

```
int main() {
    return 0;
}
```

Если main возвращает 0, значит программа завершилась нормально.

Если возвращается другое число, это считается другим кодом завершения.

## 6. Ошибки во время выполнения

Если во время выполнения произошла ошибка, VM выводит сообщение:

```
runtime error: сообщение at line N
```

После этого выполнение завершается с ненулевым кодом.

Примеры ошибок:

* деление на ноль;
* остаток от деления на ноль;
* выход за границы массива;
* неправильный сдвиг битов;
* assert не прошёл;
* panic;
* неправильное состояние VM.

Например если написать:

```
int x = 10 / 0;
```

то во время выполнения будет ошибка деления на ноль.

Для массивов тоже есть проверка:

```
int[2] a = [10, 20];
print(a[2]);
```

Индекс 2 здесь неправильный, потому что допустимы только 0 и 1.

## 7. Основные группы инструкций байткода

Инструкции можно разделить на несколько групп.

### Константы

Эти инструкции кладут значения на стек:

```
PushInt
PushUInt
PushFloat
PushBool
PushString
PushVoid
```

Например число 42 в программе превращается в инструкцию PushInt.

### Локальные переменные

Для работы с локальными переменными используются:

```
LoadLocal
StoreLocal
```

LoadLocal загружает значение переменной на стек.

StoreLocal сохраняет значение со стека в переменную.

### Арифметика

Обычные арифметические операции:

```
Add
Sub
Mul
Div
Mod
```

Они снимают значения со стека, выполняют операцию и кладут результат обратно.

### Битовые операции

У меня есть битовые операции:

```
BitAnd
BitOr
BitXor
BitNot
ShiftLeft
ShiftRight
```

Они нужны для операторов:

```
&
|
^
~
<<
>>
```

Для сдвигов есть проверка: нельзя сдвигать на отрицательное число и нельзя сдвигать на 32 или больше.

### Сравнения

Сравнения:

```
Equal
NotEqual
Less
LessEqual
Greater
GreaterEqual
```

Они возвращают bool.

Например:

```
x < y
```

после выполнения даст true или false.

### Логика и переходы

Для условий и циклов нужны переходы:

```
LogicalNot
JumpIfFalse
Jump
```

JumpIfFalse используется в if и while.

Если условие ложное, VM прыгает на другую инструкцию.

### Вызовы функций

Для функций используются:

```
Call
Return
```

Call создаёт новый кадр вызова.

Return завершает выполнение функции и возвращает результат.

### Массивы

Для массивов есть инструкции:

```
MakeArray
LoadIndex
StoreIndex
```

MakeArray создаёт массив.

LoadIndex получает элемент массива по индексу.

StoreIndex записывает элемент массива по индексу.

### Структуры

Для структур есть:

```
MakeStruct
LoadField
StoreField
```

MakeStruct создаёт структуру.

LoadField читает поле структуры.

StoreField изменяет поле структуры.

### Встроенные функции

Ещё есть встроенные операции:

```
Print
Input
Exit
Panic
Assert
Len
```

Например print в языке превращается в инструкцию Print.

len используется для строк и массивов.

## 8. Как происходит генерация кода

### Выражения

Для большинства выражений генерация идёт по стековому принципу.

Например:

```
x + y
```

генерируется примерно так:

```
загрузить x
загрузить y
выполнить Add
```

То есть сначала вычисляются операнды, потом операция.

### Присваивание

Для присваивания сначала генерируется правая часть.

Например:

```
x = a + b;
```

Сначала вычисляется a + b, результат оказывается на стеке. Потом этот результат сохраняется в x.

Если присваивание идёт в массив:

```
a[i] = value;
```

то компилятор генерирует код для массива, индекса и значения, а потом StoreIndex.

Если присваивание идёт в поле структуры:

```
p.x = 10;
```

то используется StoreField.

### if / else

Условный оператор компилируется через переходы.

Общая схема такая:

```
вычислить условие
если false, прыгнуть в else
выполнить then-часть
прыгнуть в конец
else:
выполнить else-часть
конец:
```

То есть в байткоде нет отдельной большой инструкции if. Есть обычные переходы.

### while

Цикл while тоже делается через переходы.

Схема:

```
начало цикла:
вычислить условие
если false, выйти из цикла
выполнить тело цикла
прыгнуть в начало цикла
конец цикла:
```

break превращается в переход к концу цикла.

continue превращается в переход к началу следующей итерации.

### for

for тоже можно представить через while.

Например:

```
for (int i = 0; i < 10; i = i + 1) {
    ...
}
```

работает как:

```
int i = 0;
while (i < 10) {
    ...
    i = i + 1;
}
```

В байткоде это тоже реализуется через переходы.

## 9. Вызовы функций

Аргументы функции вычисляются слева направо и кладутся на стек.

Например:

```
f(a, b)
```

Сначала вычисляется a, потом b. Потом вызывается функция f.

VM создаёт новый кадр вызова, кладёт туда параметры и начинает выполнять инструкции функции.

Когда функция делает return, результат возвращается обратно вызывающей функции.

## 10. Generic-функции

В проекте есть generic-функции.

Пример:

```
T identity<T>(T x) {
    return x;
}
```

Можно вызвать явно:

```
identity<int>(42)
```

Также поддерживается вывод типа:

```
identity(42)
```

В этом случае компилятор сам понимает, что T должен быть int.

Важно: generic-типы проверяются на этапе семантического анализа. Во время выполнения VM работает с обычными значениями Value.

То есть generic-параметры — это не отдельные значения в VM. Они нужны компилятору, чтобы проверить типы.

Моя реализация generic ближе к type erasure: компилятор проверяет типы, но не создаёт отдельную копию функции под каждый тип.

## 11. Перегрузка функций

В проекте есть перегрузка функций.

Можно написать несколько функций с одним именем, но разными параметрами:

```
int describe(int x) {
    return 1;
}

int describe(string x) {
    return 2;
}
```

При вызове компилятор смотрит на тип аргумента:

```
describe(10)
```

выберет версию для int.

```
describe("hello")
```

выберет версию для string.

Также есть перегрузка с неявным приведением для числовых типов.

Например если есть функция:

```
int onlyFloat(float64 x) {
    return 10;
}
```

то вызов:

```
onlyFloat(42)
```

разрешается, потому что int можно неявно привести к float64 при выборе перегрузки.

Точное совпадение всегда важнее приведения.

## 12. x86-64 NASM генерация

Кроме VM backend у меня есть дополнительная генерация x86-64 NASM ассемблера.

Это не главный способ выполнения программы.

Главный способ — байткод и VM.

ASM backend нужен, чтобы показать, что компилятор может генерировать более низкоуровневый код.

Он поддерживает не все возможности языка.

Например VM поддерживает больше: структуры, массивы разных типов, generic, сложные значения.

ASM backend покрывает только часть языка. Если какая-то возможность не поддержана, компилятор должен выдать ошибку, а не упасть.

После последних изменений в ASM backend добавлены битовые операции:

```
&
|
^
~
<<
>>
```

Для сдвигов в ASM тоже есть проверка неправильного количества битов.

ASM файл создаётся как:

```
out.asm
```

Запуск примерно такой:

```
./build/minilang examples/file.ml --emit-asm --no-run
nasm -f elf64 out.asm -o out.o
gcc -no-pie out.o -o out
./out
```

## 13. Режимы командной строки

У компилятора есть несколько режимов для проверки и отладки.

Можно вывести токены:

```
--dump-tokens
```

Можно вывести дерево программы:

```
--dump-ast
```

Можно вывести байткод:

```
--dump-bytecode
```

Можно сгенерировать ассемблер:

```
--emit-asm
```

Можно только скомпилировать, но не запускать VM:

```
--no-run
```

Можно вывести время фаз компиляции:

```
--time-phases
```

Это удобно для защиты, потому что можно показать не только результат программы, но и внутренние этапы компилятора.

## 14. Пример прохождения программы

Возьмём простую программу:

```
int main() {
    int x = 2 + 3;
    print(x, "\n");
    return x;
}
```

Сначала лексер превращает текст в токены.

Потом парсер строит дерево.

Потом семантический анализатор проверяет:

* есть ли main;
* правильно ли объявлены переменные;
* можно ли складывать 2 и 3;
* правильный ли тип return.

Потом генератор байткода создаёт инструкции примерно такого смысла:

```
PushInt 2
PushInt 3
Add
StoreLocal x
LoadLocal x
Print
LoadLocal x
Return
```

Потом VM выполняет эти инструкции.

В итоге программа печатает:

```
5
```

и возвращает код 5.
