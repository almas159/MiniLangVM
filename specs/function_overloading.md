# A.2.8 и A.3.1 Перегрузка функций

В MiniLangVM есть перегрузка функций.

Это значит, что можно объявить несколько функций с одинаковым именем, но с разными типами параметров.

Например:

```
int describe(int x) {
    return 1;
}

int describe(string x) {
    return 2;
}

int main() {
    return describe(10) + describe("hello");
}
```

Здесь есть две функции `describe`.

Одна принимает `int`, другая принимает `string`.

Когда компилятор видит вызов:

```
describe(10)
```

он понимает, что аргумент имеет тип `int`, значит нужно вызвать версию:

```
describe(int)
```

А когда видит:

```
describe("hello")
```

он выбирает версию:

```
describe(string)
```

## A.2.8 Перегрузка функций

Имя функции само по себе уже не всегда однозначно.

Если раньше по имени `describe` можно было найти ровно одну функцию, то теперь функций с таким именем может быть несколько.

Поэтому при семантическом анализе компилятор смотрит не только на имя функции, но и на типы аргументов.

Например:

```
int f(int x) {
    return 1;
}

int f(string x) {
    return 2;
}
```

Это нормально, потому что параметры разные:

```
int
string
```

Но вот так нельзя:

```
int f(int x) {
    return 1;
}

int f(int x) {
    return 2;
}
```

Потому что у обеих функций одинаковое имя и одинаковые типы параметров.

Компилятор выдаёт semantic error, потому что непонятно, какую функцию вызывать при:

```
f(10)
```

В моём проекте при сборе объявлений функций в таблицу функций сохраняется не одна функция по имени, а список перегрузок.

То есть условно:

```
describe -> [
    describe(int),
    describe(string),
    describe(float64)
]
```

Когда встречается вызов функции, semantic analyzer проверяет типы аргументов и ищет подходящую версию.

Если нашёл ровно одну подходящую функцию, он записывает в AST конкретный индекс функции.

После этого codegen уже не решает перегрузку заново. Он просто берёт готовый functionIndex и генерирует Call.

То есть перегрузка полностью решается на этапе semantic analysis.

## Пример обычной перегрузки

```
int describe(int x) {
    print("int: ", x, "\n");
    return 1;
}

int describe(string x) {
    print("string: ", x, "\n");
    return 2;
}

int describe(bool x) {
    print("bool: ", x, "\n");
    return 3;
}

int main() {
    int a = describe(10);
    int b = describe("hello");
    int c = describe(true);

    return a + b + c;
}
```

Здесь:

```
describe(10)
```

вызывает функцию для `int`.

```
describe("hello")
```

вызывает функцию для `string`.

```
describe(true)
```

вызывает функцию для `bool`.

Возврат будет:

```
1 + 2 + 3 = 6
```

## A.3.1 Перегрузка с неявными приведениями

Это доп 3 уровня.

Идея такая: если точной перегрузки нет, компилятор может попробовать найти функцию, куда аргумент можно безопасно привести.

В моём проекте для перегрузки поддерживаются такие неявные приведения:

```
int  -> float64
uint -> float64
```

Например:

```
int onlyFloat(float64 x) {
    return 10;
}

int main() {
    return onlyFloat(42);
}
```

Здесь функция `onlyFloat` принимает `float64`.

А мы передаём `42`, это `int`.

Точной функции `onlyFloat(int)` нет.

Но есть функция:

```
onlyFloat(float64)
```

Компилятор видит, что `int` можно привести к `float64`, выбирает эту функцию и вставляет приведение аргумента.

То есть вызов работает как будто так:

```
onlyFloat(cast<float64>(42))
```

но пользователь явно cast не пишет.

## Важное правило

Точное совпадение важнее неявного приведения.

Например:

```
int f(int x) {
    return 1;
}

int f(float64 x) {
    return 2;
}

int main() {
    return f(42);
}
```

Здесь есть два варианта:

```
f(int)
f(float64)
```

Теоретически `42` можно привести к `float64`.

Но есть точное совпадение `f(int)`, поэтому компилятор выбирает именно его.

То есть `f(42)` вернёт 1, а не 2.

Это важно, чтобы перегрузка работала предсказуемо.

## Как компилятор выбирает функцию

Примерно алгоритм такой:

1. Найти все функции с нужным именем.
2. Проверить количество аргументов.
3. Сначала попробовать найти точное совпадение типов.
4. Если точное совпадение найдено — выбрать его.
5. Если точного совпадения нет — попробовать неявные приведения.
6. Если нашлась ровно одна подходящая функция — выбрать её.
7. Если не нашлось ни одной — ошибка.
8. Если нашлось несколько одинаково подходящих — ошибка неоднозначности.

Например:

```
onlyFloat(42)
```

Точного совпадения нет.

Но есть `onlyFloat(float64)`, и `int` можно привести к `float64`.

Значит функция найдена.

А вот так:

```
int f(string x) {
    return 1;
}

int main() {
    return f(10);
}
```

Тут `int` нельзя привести к `string`.

Значит будет ошибка:

```
no matching overload for function 'f'
```

## Что хранится в AST

После выбора функции semantic analyzer записывает в узел вызова конкретный индекс функции.

То есть вызов:

```
describe(10)
```

после semantic analysis уже знает, что это не просто `describe`, а конкретная функция с конкретным functionIndex.

Это удобно, потому что генератор байткода уже не занимается поиском перегрузки.

Он просто генерирует:

```
аргументы
Call functionIndex
```

Если при выборе перегрузки нужно было неявное приведение, semantic analyzer также сохраняет целевой тип аргумента.

Потом codegen вставляет нужную инструкцию приведения, например `CastFloat`.

## Пример с неявным приведением

```
int exact(int x) {
    print("exact int\n");
    return 1;
}

int exact(float64 x) {
    print("exact float\n");
    return 2;
}

int onlyFloat(float64 x) {
    print("only float\n");
    return 10;
}

int main() {
    int a = exact(42);
    int b = exact(3.5);
    int c = onlyFloat(42);

    return a + b + c;
}
```

Что происходит:

```
exact(42)
```

Выбирается `exact(int)`, потому что это точное совпадение.

```
exact(3.5)
```

Выбирается `exact(float64)`, потому что 3.5 — это float64.

```
onlyFloat(42)
```

Точной функции `onlyFloat(int)` нет, поэтому компилятор выбирает `onlyFloat(float64)` и вставляет приведение `int -> float64`.

Итог:

```
a = 1
b = 2
c = 10
```

Возврат:

```
13
```

## Ошибки

### Одинаковая сигнатура

Так нельзя:

```
int f(int x) {
    return 1;
}

int f(int x) {
    return 2;
}
```

Ошибка будет потому, что функции полностью совпадают по имени и типам параметров.

### Нет подходящей перегрузки

Так тоже нельзя:

```
int f(int x) {
    return 1;
}

int main() {
    return f("hello");
}
```

Есть только `f(int)`, а передаётся `string`.

`string` нельзя привести к `int`, значит подходящей функции нет.

### Неоднозначность

Если компилятор найдёт несколько вариантов, которые подходят одинаково хорошо, он должен выдать ошибку неоднозначности.

Это нужно, чтобы программа не зависела от случайного порядка функций в таблице.

```
В проекте реализована перегрузка функций. Несколько функций могут иметь одно имя, если различаются списком типов параметров. На этапе semantic analysis компилятор выбирает подходящую перегрузку по типам аргументов и записывает в AST конкретный functionIndex.
```

Про A.3.1:

```
Также реализована перегрузка с неявными числовыми приведениями. Если точной перегрузки нет, semantic analyzer пробует безопасные преобразования int -> float64 и uint -> float64. Точное совпадение всегда имеет больший приоритет, чем приведение. Если подходящей функции нет или выбор неоднозначен, компилятор выдаёт ошибку.
```
