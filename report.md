# Отчёт по проекту MiniLangVM

## 1. Общее описание

MiniLangVM — это мой учебный проект по предмету “Системы программирования”.

Я сделал небольшой собственный язык программирования и компилятор для него на C++23.

Проект состоит из нескольких частей:

```
лексер
парсер
AST
семантический анализатор
генератор байткода
виртуальная машина
дополнительная генерация x86-64 NASM
тесты
спецификация языка
```

Синтаксис языка похож на C / C++ / Java. Я выбрал такой стиль, потому что он привычный и его легче объяснять.

Пример программы:

```
int main() {
    int x = 10;
    print(x, "\n");
    return 0;
}
```

---

## 2. Общий pipeline

Программа проходит такие этапы:

```
исходный код
    -> Lexer
    -> Parser
    -> AST
    -> SemanticAnalyzer
    -> BytecodeGenerator
    -> VM
```

То есть сначала текст разбивается на токены.

Потом из токенов строится дерево программы.

Потом semantic analyzer проверяет типы и смысл программы.

После этого создаётся байткод.

Потом этот байткод выполняется на моей виртуальной машине.

---

## 3. Почему основной backend — VM

Основной способ выполнения в проекте — через собственную bytecode VM.

Я выбрал VM, потому что её проще расширять.

Например строки, массивы, структуры, generic-вызовы, runtime-ошибки проще реализовывать в виртуальной машине, чем сразу в x86-64 asm.

Плюсы VM:

```
проще отлаживать
проще добавлять новые типы
проще делать runtime-проверки
проще показывать этапы компиляции
удобно делать dump bytecode
```

---

## 4. Дополнительный ASM backend

Кроме VM, в проекте есть генерация x86-64 NASM.

Это не главный backend.

Он нужен как дополнительная возможность, чтобы показать, что компилятор может генерировать низкоуровневый код.

ASM backend поддерживает только часть языка.

Основная полная поддержка языка идёт через VM.

Если ASM backend не поддерживает какую-то конструкцию, компилятор должен выдать ошибку, а не просто упасть.

---

## 5. Лексер

Лексер читает исходный текст и превращает его в токены.

Токен хранит:

```
тип токена
текст токена
позицию в файле
```

Позиция нужна для нормальных ошибок.

Лексер поддерживает:

```
ключевые слова
идентификаторы
целые числа
hex-числа
binary-числа
float-числа
exponential notation
inf
NaN
строки
операторы
скобки
однострочные комментарии
блочные комментарии
```

Примеры:

```
int x = 42;
int h = 0x2A;
int b = 0b101010;
float64 f = 1.5e2;
string s = "hello\n";
```

Если в лексере ошибка, он выдаёт диагностику.

Например:

```
незакрытая строка
незакрытый блочный комментарий
неправильный числовой литерал
```

## 6. Парсер

Парсер реализован методом рекурсивного спуска.

Он получает токены и строит AST.

Парсер поддерживает:

```
функции
generic-функции
структуры
type aliases
namespace
объявления переменных
if / else
while
for
switch
break
continue
return
print
присваивания
массивы
структуры
вызовы функций
cast
унарные выражения
бинарные выражения
```

Выражения разбираются по приоритетам.

Например:

```
2 + 3 * 4
```

понимается как:

```
2 + (3 * 4)
```

## 7. AST

AST — это абстрактное синтаксическое дерево.

Парсер строит AST, а semantic analyzer потом его проверяет и дописывает туда информацию о типах.

Основные узлы AST:

```
Program
FunctionDecl
StructDecl
TypeAliasDecl
NamespaceDecl
Statement
Expression
```

Примеры expression-узлов:

```
литералы
переменные
бинарные выражения
унарные выражения
вызовы функций
cast
массивы
структуры
индексирование
доступ к полю
```

AST нужен, чтобы следующие этапы работали уже не с текстом, а со структурой программы.

---

## 8. Семантический анализатор

Semantic analyzer проверяет смысл программы.

Он проверяет:

```
повторные объявления
использование необъявленных имён
области видимости
функции
вызовы функций
return
break / continue
присваивания
let-переменные
массивы
структуры
поля структур
операции
builtins
generic-вызовы
перегрузку функций
```

Если типы не совпадают, программа не доходит до генерации байткода.

Пример ошибки:

```
int x = "hello";
```

Это ошибка, потому что string нельзя присвоить в int.

---

## 9. Области видимости

В языке lexical scoping.

Каждый блок создаёт новую область видимости.

Пример:

```
int main() {
    int x = 10;

    {
        int y = 20;
        print(x, "\n");
        print(y, "\n");
    }

    return 0;
}
```

Внутренний блок видит `x`, потому что он объявлен снаружи.

Но снаружи переменная `y` уже не видна.

Shadowing разрешён:

```
int x = 10;

{
    int x = 20;
}
```

Но в одной области нельзя объявить две переменные с одинаковым именем.

---

## 10. Система типов

Язык статически типизированный.

Типы проверяются до выполнения.

Есть типы:

```
int / int32
uint / uint32
float64
float32 как alias
bool
string
void / unit
массивы фиксированного размера
структуры
type aliases
generic type parameters
```

`int` и `int32` — это один тип.

`uint` и `uint32` — тоже один тип.

`float32` принимается, но внутри используется общее представление float64.

`unit` — это синоним void.

Отдельного char нет. Один символ можно хранить как строку длины 1.

---

## 11. Переменные

Обычное объявление:

```
int x = 10;
```

Вывод типа через var:

```
var x = 10;
var s = "hello";
var b = true;
```

После вывода типа переменная всё равно статически типизирована.

То есть нельзя так:

```
var x = 10;
x = "hello";
```

`x` уже имеет тип int.

---

## 12. let

`let` создаёт неизменяемую переменную.

Пример:

```
let x = 10;
```

После этого нельзя написать:

```
x = 20;
```

Это semantic error.

Мутабельность — это свойство переменной, а не типа.

---

## 13. Type aliases

Можно создавать синонимы типов:

```
type Meters = int;

int main() {
    Meters d = 100;
    int x = d;
    return x;
}
```

`Meters` не является новым отдельным типом.

Это просто другое имя для `int`.

---

## 14. Массивы

Массивы имеют фиксированный размер.

Пример:

```
int[3] a = [1, 2, 3];
```

Размер входит в тип.

То есть:

```
int[3]
```

и

```
int[4]
```

это разные типы.

Поддерживается:

```
индексирование
присваивание по индексу
len
сравнение ==
сравнение !=
sizeof
```

Пример:

```
print(a[0], "\n");
a[1] = 10;
print(len(a), "\n");
```

Если индекс вышел за границы, будет runtime error.

---

## 15. Массивы разных типов

После доработок массивы могут быть не только int.

Поддерживаются:

```
int[3]
uint[3]
float64[2]
bool[2]
string[2]
Point[2]
```

Пример массива структур:

```
struct Point {
    int x;
    int y;
}

Point[2] points = [
    Point { x: 1, y: 2 },
    Point { x: 3, y: 4 }
];
```

---

## 16. Структуры

Структуры — это пользовательские типы с полями.

Пример:

```
struct Point {
    int x;
    int y;
}
```

Создание:

```
Point p = Point { x: 10, y: 20 };
```

Доступ:

```
p.x
```

Изменение:

```
p.x = 30;
```

Сравнение:

```
p == q
p != q
```

Структуры сравниваются по полям.

Типы структур номинальные. То есть структура совместима только со структурой с тем же именем.

---

## 17. Массивы внутри структур

Структуры могут хранить массивы.

Пример:

```
struct Group {
    string title;
    int[3] grades;
    string[2] names;
}
```

Использование:

```
Group g = Group {
    title: "VM group",
    grades: [90, 80, 70],
    names: ["Almas", "Bob"]
};

print(g.grades[0], "\n");

g.grades[1] = 100;
```

Это показывает, что работают вместе структуры, массивы, поля и индексирование.

---

## 18. Namespace

В языке есть namespace.

Пример:

```
namespace Math {
    int add(int a, int b) {
        return a + b;
    }
}

int main() {
    int x = Math::add(10, 20);
    return x;
}
```

namespace нужен, чтобы группировать функции и избегать конфликтов имён.

---

## 19. Generic-функции

Есть generic-функции.

Пример:

```
T id<T>(T x) {
    return x;
}
```

Вызов с явным типом:

```
int x = id<int>(42);
```

Вызов с автовыводом:

```
string s = id("hello");
```

Компилятор сам понимает, что T = string.

Также есть функции типа:

```
T choose<T>(bool cond, T a, T b) {
    if (cond) {
        return a;
    }

    return b;
}
```

Пример:

```
int x = choose(true, 10, 20);
```

Здесь T = int.

Ограничения:

```
generic structs не реализованы
bounded generics не реализованы
```

Реализация generic ближе к type erasure. То есть semantic analyzer проверяет типы, но отдельные копии функций под каждый тип не создаются.

---

## 20. Перегрузка функций

В проекте есть перегрузка функций.

Можно объявить несколько функций с одним именем, если параметры разные:

```
int describe(int x) {
    return 1;
}

int describe(string x) {
    return 2;
}
```

Вызов:

```
describe(10)
```

выберет функцию для int.

Вызов:

```
describe("hello")
```

выберет функцию для string.

Если две функции имеют одинаковые параметры, это ошибка.

Semantic analyzer выбирает нужную перегрузку и записывает в AST конкретный functionIndex.

---

## 21. Перегрузка с неявным приведением

Если точной перегрузки нет, компилятор может попробовать безопасное числовое приведение.

Поддерживаются:

```
int  -> float64
uint -> float64
```

Пример:

```
int onlyFloat(float64 x) {
    return 10;
}

int main() {
    return onlyFloat(42);
}
```

Тут 42 имеет тип int, а функция принимает float64.

Компилятор сам вставляет приведение.

Но если есть точное совпадение, оно важнее.

---

## 22. Builtins

Встроенные функции:

```
print
input
len
exit
panic
assert
sizeof
typeid
typeof
```

`print` выводит значения.

`input` читает строку.

`len` возвращает длину строки или массива.

`exit` завершает программу.

`panic` завершает программу с ошибкой.

`assert` проверяет условие.

`sizeof` возвращает размер типа.

`typeid` и `typeof` возвращают строку с типом.

---

## 23. Битовые операции

Есть битовые операции:

```
&
|
^
~
<<
>>
```

Они работают для int и uint.

Пример:

```
int a = 6 & 3;
int b = 6 | 3;
int c = 6 ^ 3;
int d = ~0;
int e = 1 << 3;
int f = 8 >> 1;
```

Для сдвигов есть проверка.

Нельзя сдвигать на отрицательное число и на число 32 или больше.

Такое даёт runtime error.

## 24. Генерация байткода

BytecodeGenerator обходит уже проверенный AST и создаёт инструкции VM.

Байткод стековый.

Например выражение:

```
x + y
```

генерируется примерно так:

```
load x
load y
add
```

Генератор байткода поддерживает:

```
литералы
переменные
арифметику
сравнения
логику
битовые операции
массивы
структуры
вызовы функций
if / while / for
builtins
```

## 25. Виртуальная машина

VM выполняет байткод.

В VM есть:

```
operand stack
call stack
stack frames
local variables
instruction pointer
function table
```

Runtime Value может хранить:

```
int
uint
float64
bool
string
array
struct
void
```

Вызов функции создаёт новый frame.

return завершает функцию и возвращает значение.

## 26. Runtime-ошибки

Runtime-ошибки выводятся так:

```
runtime error: message at line N
```

Примеры:

```
division by zero
modulo by zero
array index out of bounds
invalid shift count
assert failure
panic
invalid VM state
```

После runtime error программа завершается с ненулевым кодом.

---

## 27. Диагностика ошибок

Ошибки компиляции выводятся так:

```
file:line:column: error: message
```

Ошибки бывают:

```
лексические
синтаксические
семантические
```

Для диагностик используется `Result<T>` и `Diagnostic`.

Исключения не используются как основной способ обработки ошибок компиляции.

## 31. Ограничения

Не реализовано:

```
отдельный char
generic structs
bounded generics
pointers
nullable types
lambdas
first-class functions
overloaded operators
полноценная модульная система
IR-оптимизации
REPL
```

Перегрузка функций уже реализована, поэтому её не надо писать в ограничениях.

ASM backend поддерживает только часть языка.

Основной backend — bytecode VM.
